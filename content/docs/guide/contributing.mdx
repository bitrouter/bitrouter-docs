Thanks for your interest in contributing to BitRouter. This guide covers how to report bugs, request features, submit pull requests, and update the built-in provider registry.

## Before you start

- Read the project [README](https://github.com/bitrouter/bitrouter)
- Read the workspace architecture guide in [`DEVELOPMENT.md`](https://github.com/bitrouter/bitrouter/blob/main/DEVELOPMENT.md)
- Check existing [issues](https://github.com/bitrouter/bitrouter/issues) and [pull requests](https://github.com/bitrouter/bitrouter/pulls) before opening a new one

## Reporting bugs

Use the bug report issue template (title starts with `[BUG]`). Include:

- A clear summary of the problem
- Steps to reproduce
- Expected vs actual behavior
- Environment details (OS, Rust version, provider/config context)
- Logs, request payloads, or screenshots if helpful

If the bug is related to configuration or routing, include a minimal redacted `bitrouter.yaml` example when possible.

## Requesting features

Use the feature request issue template (title starts with `[FEATURE]`). Describe:

- The use case you're trying to solve
- Why current behavior is insufficient
- The expected UX or API shape
- Any provider-specific constraints

<Callout type="info">
For larger changes, opening an issue before writing code is the fastest way to align on approach.
</Callout>

## Submitting pull requests

1. Fork the repository and create a focused branch
2. Keep the change scoped to a single bug fix, feature, or documentation improvement
3. Add or update tests when behavior changes
4. Update docs when user-facing behavior, config, or provider support changes
5. Run validation locally before opening the PR:

```bash
cargo fmt -- --check
cargo clippy
cargo nextest run
```

6. Open a pull request with a clear summary and note any follow-up work

### What makes a good PR

- Small and easy to review
- Consistent with existing naming and crate boundaries
- Validated with the existing Rust tooling
- Accompanied by docs updates when public behavior changes

## Updating built-in providers

Built-in providers are defined as YAML files under [`bitrouter-config/providers/`](https://github.com/bitrouter/bitrouter/tree/main/bitrouter-config/providers) and loaded by `bitrouter-config/src/registry.rs`.

### Update an existing provider

Edit the matching YAML file (`openai.yaml`, `anthropic.yaml`, `google.yaml`) and update fields like `api_protocol`, `api_base`, `env_prefix`, or `models`. Then run the config and workspace tests:

```bash
cargo nextest run -p bitrouter-config
```

### Add a new built-in provider

If the provider uses an already-supported protocol:

1. Add a new YAML file under `bitrouter-config/providers/`
2. Register it in `bitrouter-config/src/registry.rs`
3. Add tests covering the registry and config loading
4. Update docs that mention supported providers

If the provider introduces a new protocol or transport, the scope is broader — plan to update:

| Crate | What to change |
|-------|----------------|
| `bitrouter-config` | Config schema and registry wiring |
| `bitrouter-runtime` | `router.rs` to instantiate the provider |
| `bitrouter-api` | Public provider-compatible endpoints |
| Workspace manifests | Feature flags for conditional compilation |

## Questions and discussion

If you want feedback before implementing a larger change, open an issue or join the community on [Discord](https://discord.gg/G3zVrZDa5C).
