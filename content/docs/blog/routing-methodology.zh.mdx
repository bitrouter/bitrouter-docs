我们对 BitRouter 的性能做出了有力的声明——93.2% 的解决率，仅为 42% 的成本。本文将详细解释我们是如何得到这些数字的、它们意味着什么，以及存在哪些局限性。

## 基准测试：SWE-bench Verified

我们使用 [SWE-bench Verified](https://www.swebench.com/) 作为主要评估基准。SWE-bench 提供来自热门 Python 仓库的真实 GitHub issue，要求 Agent 生成能够解决问题并通过仓库测试套件的补丁。

SWE-bench Verified 是完整 SWE-bench 数据集中经过人工验证的子集，包含 500 个已确认可解决且定义明确的问题。它是评估 AI 编程 Agent 的标准基准。

### 为什么选择 SWE-bench？

- **真实世界的复杂性**——这些是来自生产代码库的真实 bug 和功能需求
- **客观评估**——补丁通过运行仓库现有的测试套件来评分，而非人工判断
- **多步推理**——解决一个 issue 通常需要读取多个文件、理解代码库结构并生成正确的 diff
- **Agent 原生**——该任务天然需要多次 LLM 调用，非常适合评估路由策略

## 测试设置

### Agent 框架

我们使用标准的 Agent 循环，具备工具访问能力（文件读取、文件写入、bash 执行、搜索）。Agent 接收 issue 描述和仓库上下文，然后自主工作以生成补丁。

### 测试配置

| 配置 | 描述 | 模型池 |
|---|---|---|
| All-Opus | 每次调用使用 Claude Opus | 仅 Claude Opus |
| All-Sonnet | 每次调用使用 Claude Sonnet | 仅 Claude Sonnet |
| BitRouter Balanced | 默认路由——在保持质量的同时优化成本 | Claude Opus、Sonnet、Haiku |
| BitRouter Cost-Optimized | 激进成本优化 | Claude Opus、Sonnet、Haiku |
| All-GPT-4o | 每次调用使用 GPT-4o | 仅 GPT-4o |
| All-GPT-4o-mini | 每次调用使用 GPT-4o-mini | 仅 GPT-4o-mini |

### 评估协议

每个配置运行相同的 500 个 SWE-bench Verified 问题。我们测量以下指标：

- **解决率**——生成的补丁通过所有测试的 issue 百分比
- **每次会话成本**——完整 Agent 会话的总 LLM API 费用
- **每次会话调用数**——发出的 LLM 调用次数
- **升级率**——BitRouter 升级到更强模型的调用百分比
- **升级恢复率**——在被升级的调用中，有多少成功从较弱模型的失败中恢复

## 结果

### 解决率与成本

核心结果：BitRouter Balanced 在每次会话 $0.089 的成本下达到 93.2% 的解决率，而全 Opus 基线在每次会话 $0.212 的成本下达到 95.8% 的解决率。

这意味着解决率仅下降 2.6 个百分点，但成本降低了 58%。

### 升级分析

BitRouter Balanced 在会话中大约升级 14% 的调用。在这些升级中：

- **87%** 成功恢复——更强的模型在较弱模型失败的地方生成了正确的响应
- **9%** 部分恢复——响应有所改善但仍需进一步迭代
- **4%** 未能恢复——任务确实很难，升级也无济于事

### 路由失效的情况

BitRouter 在以下场景中不会改善（有时甚至会降低）性能：

- **单次调用任务**——如果整个任务只有一次 LLM 调用，就没有什么可以路由的
- **全面高难度任务**——某些 SWE-bench issue 每次调用都需要深度推理；路由到较弱模型只是在升级前浪费时间
- **超长会话**——超过 80 次以上调用的会话有时会随着上下文累积出现质量漂移

## 局限性与注意事项

1. **SWE-bench 仅限 Python**——我们的结果可能无法推广到其他语言或非编程任务
2. **成本取决于提供商定价**——我们的成本数据使用的是 2026 年 2 月的公开价格；实际成本因批量折扣和价格变动而异
3. **Agent 框架的影响**——不同的 Agent 架构可能获得不同的路由收益；我们的结果使用特定的 Agent 循环
4. **路由增加延迟**——分类步骤每次调用增加约 5-8ms；对于延迟敏感的应用可能有影响
5. **我们在测试自己的产品**——我们已尽最大努力保持严谨，但欢迎独立验证
