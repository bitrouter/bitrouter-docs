BitRouter plugs into OpenClaw as a native plugin. Install it, point it at your providers, and every agent request routes through a local multi-provider proxy — with failover, load balancing, and unified API access out of the box.

## Install the Plugin

```bash
openclaw plugins install @bitrouter/openclaw-plugin
```

On install, the plugin starts a BitRouter daemon on `127.0.0.1:8787`, registers `bitrouter` as an OpenClaw provider, and begins routing eligible model requests through it.

## Route Your Agents

By default, BitRouter only intercepts models with explicit routing rules. To route every request through BitRouter — including ones without a named route — set `interceptAllModels` in your OpenClaw settings:

```jsonc
{
  "plugins": {
    "entries": {
      "bitrouter": {
        "enabled": true,
        "config": {
          "interceptAllModels": true
        }
      }
    }
  }
}
```

<Callout type="info">
Even without named routes, you can use `"provider:model_id"` syntax (e.g. `"openai:gpt-4o"`) and BitRouter will direct-route it.
</Callout>

### Define Virtual Models

Expose aliases like `fast` or `smart` that fan out to multiple provider endpoints with a routing strategy:

```jsonc
{
  "plugins": {
    "entries": {
      "bitrouter": {
        "enabled": true,
        "config": {
          "models": {
            "fast": {
              "strategy": "load_balance",
              "endpoints": [
                { "provider": "openai", "modelId": "gpt-4o-mini" },
                { "provider": "anthropic", "modelId": "claude-3.5-haiku" }
              ]
            },
            "smart": {
              "strategy": "priority",
              "endpoints": [
                { "provider": "anthropic", "modelId": "claude-sonnet-4-20250514" },
                { "provider": "openai", "modelId": "gpt-4o" }
              ]
            }
          }
        }
      }
    }
  }
}
```

Strategies: `priority` (default — try endpoints in order, fall over on failure) and `load_balance` (round-robin).

## Verify

```bash
bitrouter status
```

You should see the daemon marked as **healthy**, its listen address, and the registered providers. Agents can call the same check at runtime via the `bitrouter_status` tool.

## What You Get

- **Failover & load balancing** — priority or round-robin routing across providers. [Learn more](/docs/overview/runtime-optimization)
- **Virtual models** — aliases like `fast` or `smart` that map to multiple provider endpoints with routing strategies
- **Custom providers** — derive from `openai`, `anthropic`, or `google` to add providers like OpenRouter with a few lines of config
- **Usage metrics** — per-model, per-provider request tracking
- **Agent key management** — scoped Ed25519 keypair and JWT tooling via the `bitrouter_keygen`, `bitrouter_account`, and `bitrouter_keys` agent tools
