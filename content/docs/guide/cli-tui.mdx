BitRouter ships a full CLI for managing providers, routes, accounts, and daemon lifecycle. Running `bitrouter` with no arguments launches an interactive TUI dashboard.

## Server lifecycle

### Foreground server

```bash
bitrouter serve
```

Starts the API server in the foreground with logs printed to the terminal. Useful for development.

### Background daemon

```bash
bitrouter start
```

Runs BitRouter as a background daemon. The PID is stored in `~/.bitrouter/run/bitrouter.pid` and logs go to `~/.bitrouter/logs/`.

```bash
bitrouter stop       # gracefully stop the daemon
bitrouter restart    # stop then start
```

### Status

```bash
bitrouter status
```

Prints daemon PID, home directory, listen address, configured providers, and active routes.

## Setup wizard

```bash
bitrouter init
```

Interactive setup that walks you through provider selection and API key entry. Generates `bitrouter.yaml` and `.env` in the BitRouter home directory. Also triggered automatically on first TUI launch when no providers are configured.

<Callout type="info">
The wizard auto-detects API keys already in your environment (e.g. `OPENAI_API_KEY`, `ANTHROPIC_API_KEY`) and offers to use them.
</Callout>

## Account management

BitRouter uses Ed25519 keypairs for JWT signing and on-chain identity.

```bash
bitrouter account -g            # generate a new keypair
bitrouter account -l            # list all keypairs
bitrouter account --set <ID>    # set active account by index or pubkey prefix
```

Each account has a Solana address, EVM address, and public key prefix. Keys are stored in `~/.bitrouter/.keys/`.

## Token generation

Create JWTs signed by the active account:

```bash
bitrouter keygen --scope api --exp 30d --models "openai:*,anthropic:claude-*"
```

| Flag | Description |
|------|-------------|
| `--chain <CHAIN>` | Signing chain: `solana` (default) or `base` |
| `--scope <SCOPE>` | Token scope: `api` (default) or `admin` |
| `--exp <EXP>` | Expiration: `5m`, `1h`, `30d`, `never` |
| `--models <MODELS>` | Comma-separated model patterns |
| `--tools <TOOLS>` | Comma-separated tool patterns (e.g. `github/*,jira/search`) |
| `--budget <BUDGET>` | Budget limit in micro USD |
| `--budget-scope <SCOPE>` | `session` or `account` |
| `--budget-range <RANGE>` | e.g. `rounds:10`, `duration:3600s` |
| `--name <LABEL>` | Save the token locally with a label |

<Callout type="info">
Admin tokens require an explicit `--exp` value to limit the replay window.
</Callout>

### Manage saved tokens

```bash
bitrouter keys -l              # list saved tokens
bitrouter keys --show <ID>     # show decoded JWT claims
bitrouter keys --rm <ID>       # remove a saved token
```

## Route management

Manage model routes at runtime (requires a running daemon):

```bash
bitrouter route list
bitrouter route add my-model openai:gpt-4o anthropic:claude-sonnet-4-20250514 --strategy priority
bitrouter route rm my-model
```

| Strategy | Behavior |
|----------|----------|
| `priority` | Use the first endpoint; failover to the next on error |
| `load_balance` | Round-robin across endpoints |

## Inspect upstreams

### MCP tools

```bash
bitrouter tools list      # list all available MCP tools
bitrouter tools status    # show upstream MCP server health
```

### A2A agents

```bash
bitrouter agents list     # list configured upstream agents
bitrouter agents status   # show connection health
```

## Global flags

These flags are available on every command:

| Flag | Description |
|------|-------------|
| `--home-dir <PATH>` | Override BitRouter home directory |
| `--config-file <PATH>` | Override `<home>/bitrouter.yaml` |
| `--env-file <PATH>` | Override `<home>/.env` |
| `--run-dir <PATH>` | Override `<home>/run` |
| `--logs-dir <PATH>` | Override `<home>/logs` |
| `--db <DATABASE_URL>` | Override database connection URL |

### Home directory resolution

BitRouter resolves the home directory in this order:

1. `--home-dir` flag
2. `./bitrouter.yaml` in the current working directory
3. `BITROUTER_HOME` environment variable
4. `~/.bitrouter` (default, auto-created if missing)

## TUI dashboard

```bash
bitrouter
```

Running BitRouter with no arguments launches the interactive TUI. It displays:

- Daemon status (running with PID, or stopped)
- Listen address
- Number of configured providers
- Number of active routes

The TUI adapts its layout to terminal width, showing a full ASCII logo on wide terminals and a compact version on narrow ones.

### Keyboard shortcuts

| Key | Action |
|-----|--------|
| `q` | Quit |
| `Ctrl+C` | Quit |

### First-run behavior

If no providers are configured, the TUI automatically launches the setup wizard (`bitrouter init`). After completing setup, the TUI reloads with the new configuration.

<Callout type="info">
The TUI is an optional feature enabled by default. Build without it using `cargo build --release --no-default-features`. See [Installation](/docs/guide) for details.
</Callout>
