如今每个 AI Agent 都面临同样的问题：你选定一个模型，然后将所有调用都发送给它。选择最好的模型，你会在廉价模型就能完成的任务上浪费资金。选择便宜的模型，你的 Agent 在遇到困难任务时就会卡住。

BitRouter 解决了这个问题。它位于你的 Agent 和 LLM 提供商之间，实时监控每个会话，将每次调用路由到能够处理该任务的最便宜模型——当出现问题时自动升级到更强大的模型。

## Agent 工作负载中的成本问题

Agent 会话不是单次提示。一个典型的编程 Agent 会话可能涉及 20-50 次 LLM 调用：读取文件、规划变更、编写代码、运行测试、修复错误。其中大多数调用都很简单——Agent 在读取文件或做小幅修改。只有少数调用需要深度推理。

然而如今，整个会话都运行在一个模型上。如果你为了质量选择 Claude Opus 或 GPT-4o，那么每一次调用都在支付顶级价格，包括那些简单的调用。

我们对数千个 Agent 会话的分析表明，**会话中 60-70% 的调用可以由更小、更便宜的模型处理**，而不会影响结果质量。

## BitRouter 的工作原理

BitRouter 是一个即插即用的代理。你只需修改 Agent 配置中的一行——替换 base URL——其他一切保持不变。

```bash
# Before
OPENAI_BASE_URL=https://api.openai.com/v1

# After
OPENAI_BASE_URL=https://api.bitrouter.ai/v1
```

在这个 URL 背后，BitRouter 会：

1. 根据提示内容、对话历史和任务上下文**分类**每个传入调用
2. **路由**到满足难度阈值的最便宜模型
3. 实时**监控**响应质量
4. 如果响应质量下降或 Agent 进入错误循环，**升级**到更强大的模型

整个路由决策在 p99 下增加不到 10ms 的延迟。

## SWE-bench 结果

我们在 [SWE-bench Verified](https://www.swebench.com/) 上评估了 BitRouter，这是 AI 编程 Agent 的标准基准测试。我们的均衡配置达到了：

- **93.2% 解决率**（对比全 Opus 基线的 95.8%）
- 仅为最强模型全量运行**成本的 42%**
- **87% 升级恢复率**——当便宜模型失败时，升级到更好的模型有 87% 的概率能恢复任务

成本优化配置进一步降低了开支——仅为基线成本的 28%——同时保持 89.1% 的解决率。

## 快速开始

BitRouter 可与任何 OpenAI 兼容的 Agent 框架配合使用。设置不到一分钟：

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "https://api.bitrouter.ai/v1",
  apiKey: process.env.BITROUTER_API_KEY,
});
```

就是这样。你现有的 Agent 代码、提示词和工具定义无需任何更改即可正常使用。
