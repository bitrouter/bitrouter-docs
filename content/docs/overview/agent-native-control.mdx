AI agents operate on-demand with unpredictable usage patterns. Traditional SaaS subscriptions, prepaid credits, and API keys don't fit. BitRouter replaces API keys with wallet signatures — agents authenticate and pay per-use directly from a self-custodial wallet, within boundaries you define.

## Wallet-Based Access

Instead of issuing API keys, BitRouter authenticates every request with a wallet signature. Payments are processed per-use via the [x402 protocol](https://www.x402.org/) on Base or Solana — no custodial intermediaries, no prepaid balances, no vendor lock-in.

### Supported Pricing Models

- **Per-call pricing** — fixed cost per API request
- **Token-based pricing** — cost based on input/output tokens (for LLMs)

## Payment Delegation Key (PDK)

To let an agent act on your behalf, you create a Payment Delegation Key. A PDK grants an agent delegated spending authority from your wallet — without exposing your private key. You define the constraints, and the agent operates autonomously within them.

### Permission Parameters

| Parameter | Type | Description |
|-----------|------|-------------|
| `max_transaction_value` | number | Maximum value allowed per single transaction |
| `daily_spending_limit` | number | Total amount that can be spent within 24 hours |
| `valid_until` | timestamp | When the PDK expires and becomes invalid |
| `allowed_recipients` | address[] | Addresses the PDK can send transactions to |

For most use cases, setting a `daily_spending_limit` and `valid_until` provides sufficient protection while allowing agents to operate autonomously.

<Callout type="info">
The current PDK implementation is built on [Privy's policy engine](https://docs.privy.io/controls/overview) and is self-custodial. Future versions will be signer-compatible, allowing direct use with any external wallet. For advanced configurations like contract allowlists, calldata restrictions, and typed data signing rules, see the [Privy documentation](https://docs.privy.io/controls/policies/overview).
</Callout>
