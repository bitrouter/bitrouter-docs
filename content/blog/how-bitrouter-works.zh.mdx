BitRouter 位于你的 AI Agent 和 LLM 提供商之间。它看起来像一个标准的 OpenAI 兼容 API，但在幕后，它对每次调用都做出智能路由决策。本文将详细介绍其技术实现。

## 架构概览

```
Agent → BitRouter Gateway → Router → LLM Provider
                              ↑
                         Session Store
```

网关接收标准的 chat completion 请求。在转发给 LLM 提供商之前，路由器会检查请求并决定使用哪个模型。Session Store 在同一 Agent 会话的多次调用之间维护上下文。

## 路由管线

每个传入请求经过四个阶段：

### 1. 会话识别

BitRouter 使用多种信号将相关调用归组为会话：

- 显式的 session ID（如果你的 Agent 提供了的话）
- 对话线程检测（跨调用匹配 `messages` 数组）
- 时间邻近性（来自同一 API key 的短时间窗口内的调用）

会话归组至关重要，因为路由决策依赖于会话中之前发生的事情。单独看起来简单的调用，可能是需要更强模型的错误恢复循环的一部分。

### 2. 任务分类

分类器检查传入请求并分配难度评分。它考虑以下因素：

- **提示复杂度**——长度、指令数量、是否包含代码、结构化输出要求
- **工具使用模式**——Agent 最近调用了哪些工具，结果如何
- **会话轨迹**——Agent 是否在取得进展，还是陷入了循环？
- **错误信号**——测试失败、语法错误，或最近几轮中 Agent 的自我修正

分类器本身是一个轻量级模型——一个微调的嵌入模型，运行时间不到 2ms。它输出 0 到 1 之间的难度评分。

### 3. 模型选择

难度评分通过可配置的阈值表映射到模型层级：

| 难度 | 均衡配置 | 成本优化配置 |
|---|---|---|
| 0.0 – 0.3 | Haiku | Haiku |
| 0.3 – 0.6 | Sonnet | Haiku |
| 0.6 – 0.8 | Sonnet | Sonnet |
| 0.8 – 1.0 | Opus | Sonnet |

这些阈值是可调的。均衡配置偏重质量（对中等任务使用 Sonnet），而成本优化配置将更多调用推向 Haiku。

### 4. 响应监控与升级

将请求发送到选定模型后，BitRouter 会监控响应的质量信号：

- **连贯性检查**——响应是否在逻辑上符合提示？
- **完整性检查**——模型是否完成了响应还是被截断了？
- **错误模式检测**——响应是否包含已知的失败模式（如"I don't know"、重复内容、格式错误的代码）？
- **工具调用验证**——如果模型在调用工具，调用格式是否正确？

如果质量信号指示存在问题，BitRouter 会触发**升级**：将同一请求重新发送到更强大的模型。升级后的响应将替换原始响应。

升级决策在实时发生，通常在流式响应的前几个 token 内完成。这最大限度地减少了计算浪费——我们不会等到完整响应后才决定是否升级。

## 会话感知

BitRouter 中最重要的架构决策是维护会话状态。没有它，路由器将独立处理每次调用，错过关键上下文。

### 会话状态追踪的内容

- **调用历史**——之前的调用使用了哪些模型，结果如何
- **错误累积**——会话中发生了多少错误，以及最近的频率
- **升级历史**——哪些调用被升级了，升级是否有帮助
- **任务阶段检测**——Agent 当前是处于探索、实现还是调试阶段

### 会话状态如何影响路由

一个简单的例子：如果最近三次调用都产生了错误，路由器会偏向为下一次调用选择更强大的模型，无论分类器的结果如何。反之，如果会话已经在 Haiku 上顺利运行了 10 次调用，就没有理由升级。

这种会话感知的方法是 BitRouter 优于静态路由规则的原因。固定的"短提示用 Haiku，长提示用 Opus"的规则忽略了 Agent 会话的动态特性。

## 延迟预算

BitRouter 为每次调用增加了开销。时间分布如下：

| 组件 | 延迟 (p50) | 延迟 (p99) |
|---|---|---|
| 会话查找 | 0.5ms | 1.2ms |
| 任务分类 | 1.5ms | 2.8ms |
| 模型选择 | 0.1ms | 0.3ms |
| 请求转发 | 2.0ms | 5.0ms |
| **总开销** | **4.1ms** | **9.3ms** |

总开销在 p99 下保持在 10ms 以内。作为对比，LLM 响应本身通常需要 500ms-10s，具体取决于模型和提示长度。BitRouter 的开销可以忽略不计。

我们正在积极开发：

- **多提供商路由**——在单个会话中根据每个提供商的优势，跨 Anthropic、OpenAI 和 Google 模型进行选择
