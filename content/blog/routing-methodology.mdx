We make strong claims about BitRouter's performance — 93.2% resolve rate at 42% of the cost. This post explains exactly how we got those numbers, what they mean, and where the limitations are.

## Benchmark: SWE-bench Verified

We use [SWE-bench Verified](https://www.swebench.com/) as our primary evaluation benchmark. SWE-bench presents real GitHub issues from popular Python repositories and asks the agent to produce a patch that resolves the issue and passes the repository's test suite.

SWE-bench Verified is a human-validated subset of the full SWE-bench dataset, containing 500 problems confirmed to be solvable and well-specified. It's the standard benchmark for evaluating AI coding agents.

### Why SWE-bench?

- **Real-world complexity** — these are actual bugs and feature requests from production codebases
- **Objective evaluation** — patches are graded by running the repository's existing test suite, not by human judgment
- **Multi-step reasoning** — solving an issue typically requires reading multiple files, understanding the codebase structure, and producing a correct diff
- **Agent-native** — the task naturally requires multiple LLM calls, making it ideal for evaluating routing strategies

## Test setup

### Agent framework

We use a standard agent loop with tool access (file read, file write, bash execution, search). The agent receives the issue description and repository context, then works autonomously to produce a patch.

### Configurations tested

| Configuration | Description | Model pool |
|---|---|---|
| All-Opus | Every call uses Claude Opus | Claude Opus only |
| All-Sonnet | Every call uses Claude Sonnet | Claude Sonnet only |
| BitRouter Balanced | Default routing — optimize cost while preserving quality | Claude Opus, Sonnet, Haiku |
| BitRouter Cost-Optimized | Aggressive cost optimization | Claude Opus, Sonnet, Haiku |
| All-GPT-4o | Every call uses GPT-4o | GPT-4o only |
| All-GPT-4o-mini | Every call uses GPT-4o-mini | GPT-4o-mini only |

### Evaluation protocol

Each configuration runs the same 500 SWE-bench Verified problems. We measure:

- **Resolve rate** — percentage of issues where the generated patch passes all tests
- **Cost per session** — total LLM API cost for the full agent session
- **Calls per session** — number of LLM calls made
- **Escalation rate** — percentage of calls where BitRouter upgraded to a more capable model
- **Escalation recovery rate** — of escalated calls, how many recovered from the cheaper model's failure

## Results

### Resolve rate vs. cost

The headline result: BitRouter Balanced achieves 93.2% resolve rate at $0.089 per session, compared to the all-Opus baseline at 95.8% resolve rate and $0.212 per session.

That's a 2.6 percentage point drop in resolve rate for a 58% cost reduction.

### Escalation analysis

BitRouter Balanced escalates approximately 14% of calls within a session. Of those escalations:

- **87%** successfully recover — the more capable model produces a correct response where the cheaper model failed
- **9%** partially recover — the response improves but still requires further iterations
- **4%** don't recover — the task is genuinely hard and the escalation doesn't help

### Where routing fails

BitRouter does not improve (and sometimes hurts) performance on:

- **Single-call tasks** — if the entire task is one LLM call, there's nothing to route
- **Uniformly hard tasks** — some SWE-bench issues require deep reasoning on every call; routing to cheaper models just wastes time before escalating
- **Very long sessions** — sessions exceeding 80+ calls sometimes see quality drift as context accumulates

## Limitations and caveats

1. **SWE-bench is Python-only** — our results may not generalize to other languages or non-coding tasks
2. **Cost depends on provider pricing** — our cost numbers use list prices as of February 2026; actual costs vary with volume discounts and pricing changes
3. **Agent framework matters** — different agent architectures may see different routing benefits; our results use a specific agent loop
4. **Routing adds latency** — the classification step adds ~5-8ms per call; for latency-sensitive applications this may matter
5. **We're benchmarking our own product** — we've made every effort to be rigorous, but independent verification is welcome
